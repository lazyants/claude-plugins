# The fix can be worse than the bug

A fix to an AMBIGUOUS heuristic can be a NET-NEGATIVE — trading a benign/visible failure for a severe/silent one. A reviewer flag "X isn't handled" makes the reflex "add a gate/heuristic," but a gate keyed on an unreliable signal misfires in the OTHER direction, and its failure mode can be far worse (silent / data-losing) than the one it fixed. An ambiguous heuristic has NO clean fix — each patch spawns a new edge.

Concrete shape: an unconditional flatten promoting a front-matter `<h2>` (TOC) to a spurious chapter is a visible, content-preserved failure; gating the flatten on a classifier that returns `regenerate` for any `<img>` then drops a real illustrated chapter to a PLACEHOLDER — silent data loss. The fix was strictly worse than the bug; a REVERT was the answer.

**How to apply:**

- Before fixing a reviewer's edge-case flag with a new gate/heuristic, ask: **"what is the worst failure of the FIX ITSELF, and is it worse — especially silent or data-losing — than the failure it fixes?"** A fix that trades a visible/recoverable failure for a silent/lossy one is a regression even though it closes the ticket. Compare failure MODES, not just "does it fix the reported case."
- On a genuinely ambiguous classification, **err toward the failure direction that is recoverable and VISIBLE** (over-flatten → a spurious but human-visible chapter) over the silent-lossy one (under-flatten → placeholder).
- When successive review rounds keep surfacing new edge cases in the SAME heuristic (each fix spawning the next), that is the signal the problem is fundamentally ambiguous → **STOP patching and ESCALATE the tradeoff to the user** via AskUserQuestion with a concrete recommendation + the failure-mode asymmetry + a documented/tested residual, rather than iterate. Pre-commit to a round-N escalation trigger so you actually pull the ripcord.

**Plan-clean is not fix-correct.** An ambiguous-widen fix can pass 3 rounds of PLAN-level codex review (which checks internal design consistency, not empirical behavior) and still break a documented, deliberate design decision — only code-level adversarial review running the actual change against real inputs catches it. When the domain has known fixed-form exceptions, resist the ad-hoc allowlist (whack-a-mole) and escalate the same way: offer the real options (revert vs. build a curated exception mechanism) with the failure-mode asymmetry.

**Removing a verification shifts the risk onto whoever consumed it.** Narrowing/loosening a check to kill a false-positive can REMOVE the only binding a downstream consumer relied on (e.g. dropping free-text bodies from a verdict compare removes the guarantee the fixer applied the reviewer's real guidance, so a substantive mis-transcription now passes). Field-PRESENCE checks (plan review, codex) miss content-divergence regressions; a pedantic review-bot catches them. When you remove or loosen a verification to fix a false-positive, ask what DEPENDED on that verification and re-establish the guarantee via a DIFFERENT path (redirect the consumer to read the authoritative, token-validated on-disk artifact), not just delete it. Read the issue's own net-negative caveats — under-reading them is how the regression slips past plan review.
