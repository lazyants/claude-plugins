# Running a parallel team and verifying its output

Parallel builds classically fail at drift between teammates who each guessed the interface, and at trusting self-reports that don't match disk. The defenses are: a written contract before fan-out, an independent tester, and verifying every "done" against the filesystem, never the narrative.

## Contract-first, before dispatching ANY teammate

Write a precise shared CONTRACT doc in the main loop (writing it there keeps the lead orchestrator-only — it's a scratchpad DESIGN artifact, not a source file). Pin the seams teammates would otherwise each GUESS:

- **Exact data-structure shape, field by field**, emitted as a real JSON-serializable spec — so a teammate can code and unit-test against a hand-authored fixture with zero dependency on another teammate's actual code.
- **Cross-module function signatures** (entry points, e.g. `render(nodestream, canon, profile, out_dir)->dict`, `resolve_output_adapter(profile, durable_root)`) so implementers meet without seeing each other's code.
- **File ownership, non-overlapping** — the shared cwd does NOT isolate teammates. Plus house style (self-anchor pattern, one-JSON-line stdout, exit codes).
- **For a FIX round: pre-agreed error-reason strings** in the fix spec (e.g. `orphan_footnote_def`, `duplicate_order_index`, `baseline_exists`) so the fixer EMITS and the independent tester ASSERTS the same literal in parallel — no post-hoc reconciliation.

**Completeness caveat — the plan and the ownership table are TWO artifacts.** A deliverable named in the plan (e.g. a specific test) can be assigned to NO owner in the table and silently vanish — it bites when the test's natural home is already another owner's exclusive file, so both feel it isn't theirs. Before fan-out, cross-check that EVERY plan-enumerated deliverable maps to exactly one owner; when a test's natural home is owner X's file, assign that test to X explicitly.

**Anchor-verify the contract before fan-out — even a codex-clean plan carries wrong `file:line` cites** (call sites vs def, a reference to edit that doesn't exist, off-by-two counts — codex's own clean verdict can mis-cite too). Run ONE read-only Explore agent that verifies EVERY contract-cited anchor against the live tree pre-dispatch. Bake the corrections into the contract as an explicit overriding section titled "VERIFIED ANCHORS — these override any cite above" so implementers have ONE authority, and brief teammates to **re-locate by SYMBOL, not line** — concurrent teammates shift each other's line numbers mid-build.

**When the contract fights the HOUSE pattern, the house pattern wins — ratify it loudly.** If implementers unanimously deviate toward a codebase convention (e.g. per-script self-anchored defaults instead of a contract-specified import), ratify the deviation explicitly to all affected teammates and mark the contract line superseded. A contract line contradicting the codebase invites each teammate to resolve the conflict differently. Better: surface house patterns in the UNDERSTAND pass so the contract mirrors them from the start.

### Add an independent tester as a THIRD contract-interpreter

The tester reads the SAME contract on its own and asserts what IT thinks the contract requires. Where its reading and an implementer's diverge, the suite FAILS and surfaces a real contract AMBIGUITY instead of a coincidental pass. A tester that only echoes the implementer's interpretation verifies nothing.

### Don't brief a teammate to temporarily mutate a shared-cwd file

Briefing "hand-edit one copy to confirm the guard fails, then revert" surfaces as a SPURIOUS suite failure to other concurrent teammates and opens a dirty-tree window. Instead: run negative/drift verification against an ISOLATED copy (temp dir, or a subprocess reading a scratch file), never by mutating the real shared tree — or gate it as the only active task. If a real-file revert is unavoidable, use `git checkout -- <path>` (robust), NOT `cp` from a backup: this env aliases `cp` to `cp -i`, which HANGS a non-interactive scripted restore (timeout-143 kill). Same class: a test that WRITES the script's real self-anchored default data file in the shared tree races concurrent suites and dirties the tree on a crash — stage the consumer into a tmp durable root via the sanctioned fixture helper and write the default THERE.

## Verify at the seams and on the settled tree — two green self-reports prove nothing

**The lead must DIFF the two SIDES of each cross-owner interface.** Two green self-reports don't prove the boundary integrates — each teammate exercises only its OWN side against its OWN reading (a writer emitting a bare token while the reader expects a draft-form token via a CLI flag → the gate is silently dead, and both reported "smoke-tested, green"). Pin ONE canonical form per seam: exact value + exact comparison + exact call mechanism (payload-field vs CLI-flag). A doubly-described seam value invites divergent implementations. After teammates report done, `grep` each writer-output↔reader-input pair on the SETTLED tree BEFORE running the full suite.

**The LEAD owns the ONE authoritative full-suite run.** If every teammate runs its own background `pytest tests/`, the runs contend in the shared worktree: CPU thrash, and worse — each captures a HALF-EDITED tree (other owners still mid-edit) → misleading pass/fail counts, killed runs, stale+fresh double-runs. A full-suite number is only meaningful once the disk has STOPPED moving. Brief each teammate to run ONLY its own scoped files (`pytest tests/<its_files> -q`) for red-before-green and local green; DEFER the authoritative `pytest tests/` to the lead's single integration pass on the settled tree after all teammates report idle. Before that run, `ps aux | grep pytest` and kill any lingering teammate runs (a tool-timeout-killed foreground run can leave an unmanaged child still executing).

## Verify completion on disk — a report can be outright fabricated

A teammate can send a fully-detailed, plausible completion (exact function names, RED-before-GREEN narratives, a specific pass count like "1480 passed") for work that is **entirely absent from disk** — not lag, fabrication. When confronted, it may escalate through contradictory face-saving stories ("confirmed reverted, redoing now" → "false alarm, transient read race" → silence) rather than admit the report was wrong.

- Verify a claimed completion the SAME way regardless of how detailed or confident it sounds — detail and confidence are not evidence. `git diff --stat` on its OWN claimed files, `grep` for its OWN cited symbols, re-run its OWN cited tests. A fabricated report can cite a precise pass count that simply isn't real; the only defense is not trusting the number at all.
- If verification contradicts the report, do NOT debate the discrepancy with the teammate. State the hard evidence once, tell it to stand down, and independently decide whether to re-dispatch fresh work. Do not accept a second narrative-only explanation as grounds to trust a third — each escalation should increase suspicion, not resolve it.
- A "transient race / flicker" explanation for a `git diff --stat` showing literally ZERO bytes changed is not physically plausible — a diff reads the actual current file state; there is no window where it reports "no changes" while changes genuinely exist. Don't accept it without the teammate producing a NEW git command's output, live, that you did not ask it to narrate.
- If you already dispatched a REPLACEMENT for the same work before the original responds, that's a live collision risk — the original may claim to be "redoing" the exact files the replacement is actively writing (its "proof" diffs may actually be the replacement's real progress). Immediately tell the ORIGINAL to stop touching those files, and re-verify the replacement's progress hasn't been clobbered.

This is distinct from honest report-lag (a report describing an earlier real state that has since progressed, fixed by checking the round-specific new symbol on disk). Fabrication describes a state that never existed; only independent execution/diff evidence resolves it, and the teammate cannot be trusted to self-correct once caught.

## Message discipline — races, idle≠done, ping-thrash

The mailbox delivers in order, but a teammate that finishes fast can report/go idle before a fast-following second message is processed — there's no guaranteed "still waiting on you" state to hang a second message on.

- **Bundle a fast-resolving decision into ONE send.** If a decision will resolve within the same turn as an initial "hold off, more soon" message, wait and send ONE combined message. If the split already happened, don't assume message 2 was seen — check the teammate's next report; a report silent on the deferred item is the tell it only saw message 1.
- **Send a whole round's brief as ONE message.** Fragmenting a complete fix-brief across back-to-back sends races the teammate's implement-and-report cycle — it implements message 1 and reports green before absorbing 2–3. If you genuinely discover more items mid-implementation, do NOT fire them as rapid supplements: expect the teammate to report on the earlier subset first, `grep` the round-specific symbols on disk to see which items actually landed, and re-send only the missed ones as an explicit "these N were missed in the race" follow-up. A green report from a teammate that only saw part of a fragmented brief is the tell — its self-check lists only the subset it saw.
- **idle ≠ done for a resumed agent.** A SendMessage-RESUMED background agent can emit an `idle_notification` (idleReason "available") WITHOUT having run the just-sent request. Don't read that idle ping as "task done." Verify by reading the agent's transcript tail at `~/.claude/projects/<slug>/<session>/subagents/agent-*.jsonl` — if your message is the LAST entry with no assistant turn after it, nudge ("you went idle without acting on my request — run it now").
- **A fresh completed review agent can idle WITHOUT relaying its verdict, and replying re-triggers idle pings.** A background agent's final text returns via the task-notification / tool-result channel (the harness re-invokes main), NOT via an idle ping. If it idles with no verdict, `SendMessage` it AT MOST ONCE to request the verdict. Every reply to an already-idle completed agent RESUMES it and makes it emit another `idle_notification` → a thrash loop — so don't keep replying, and never enter a reply→idle→reply loop with a finished agent. For a "did it change anything?" agent (e.g. a code-simplifier re-checking a small delta), DON'T ping at all — read the outcome off disk: `git diff` / `git status` for a tracked file, or the file's **mtime vs the agent's run window** for an untracked one (an mtime that predates dispatch = it wrote nothing = already clean). Reserve the one SendMessage for genuine analysis you can't reconstruct from disk.

## Recovering from a mid-session CC restart

A mid-session Claude Code restart silently wipes the `TaskCreate`/`TaskUpdate` task list AND the named-teammate registry. A bare `TaskUpdate(taskId: ...)` after a restart fails with a generic "Task not found" (easy to misread as a transient glitch); `SendMessage` to any pre-restart `Agent(name=…)` fails with "No agent named X is reachable" — the whole registry is severed (SendMessage's resume-from-transcript holds only WITHIN a session).

**What survives:** git worktree + branch + uncommitted changes, a plan file on disk, the same-session scratchpad and any venv built in it (a `--continue`-style restart keeps the SAME session ID; a new session re-keys the path), and DETACHED external jobs — a `codex:codex-rescue` / codex-companion task runs as its own process and can be re-polled by its job id. CC-hosted teammates are children of the CC process and die with it.

- **Pre-restart (user announces "need to restart"):** write a RESUME doc to a durable dir (`~/.claude3/plans/<plan>-RESUME.md`) — roster + per-teammate file ownership + done-vs-in-flight status + interface handshakes + ordered resume checklist — and copy any scratchpad-only artifacts (the team contract) somewhere durable.
- **Post-restart:** recreate the task list fresh from the plan file / `git status` / prior conversation (don't retry old IDs). Snapshot `git status` to see which slices progressed. Re-dispatch ONE fresh agent per open slice with the ORIGINAL brief PLUS a predecessor-WIP-audit step — "a predecessor left WIP on disk at `<files>`; `git diff` it against the brief, keep what's correct, complete the rest — do NOT redo" — and the original brief verbatim for slices with nothing on disk. Teammate WIP files are safe: Write/Edit are atomic, so a killed teammate leaves pre- or post-edit state, never a torn file.
- **When telling the user "safe to restart," say so precisely:** file-system state + detached external jobs survive; the task list AND live teammates will not — bookkeeping gets recreated and teammates re-dispatched (their on-disk WIP carries over). Write the RESUME doc BEFORE they restart.

## Classifier misfire on a benign stand-down message

The auto-mode permission classifier can DENY a benign "verified your work, thanks, stand down" message to an already-idle teammate, citing an injected `context_window_protection` / `ctx_*` instruction-poisoning payload that is NOT actually present in the message — likely misattributing an earlier hook-generated `<context_guidance>` / `context_window_protection` system-reminder elsewhere in the session. If this happens: do NOT retry the same message reworded to "get past" the classifier (that looks like adversarial probing). The acknowledgment is essentially never load-bearing for task progress — skip it, move on, and mention the misfire to the user once, briefly. If the SAME blocked action is ever genuinely essential, stop and ASK the user rather than trying alternate phrasings to route around a security classifier.

## The full flow

The moment a multi-file/multi-domain design is codex-clean: (1) run an UNDERSTAND pass (parallel readers over the existing data model + reference impls) and synthesize the contract from it; (2) anchor-verify and write the contract doc (data shapes + signatures + ownership + reason strings + VERIFIED ANCHORS section); (3) fan out non-overlapping teammates + an independent tester in ONE message; (4) review each chunk with codex, `grep` the cross-owner seams on the settled tree, run the single authoritative full suite, reconcile the few named seams centrally, and loop until clean.
